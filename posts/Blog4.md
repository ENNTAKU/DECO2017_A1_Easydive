This week, I started thinking about our platform in a more technical way. Up to now, I had mostly been thinking about user needs, page flow, and what features might be useful. But once I tried to connect those ideas to the kind of web development I am actually learning now, I realised I needed to be much more specific. A feature only becomes realistic when I can imagine how the data is entered, handled by the backend, and shown again on the page.
At the moment, my technical learning is still quite basic. I am mainly working with HTML and Java for backend web development, so I need to think about the project in a way that matches that level. That means I feel more confident with things like forms, structured input, simple server-side handling, and displaying stored information back to the user. I do not think it would make sense for me to design around highly advanced systems that depend on real-time updates, intelligent recommendations, or complicated matching logic, because those are not things I can build well right now.
This became clearer when I thought about the user profile feature. On the surface, it sounds simple, but it still needs clear structure. If I imagine it in a way that fits my current skills, a user profile would probably include fields like username, diving experience, gender if the user chooses to provide it, a short self-introduction, and a few preset diving-related tags. I think preset tags are a better choice than free text because they are easier to handle in HTML forms and easier to process on the Java backend. They also make filtering or displaying profile information much cleaner later on.
The same kind of thinking applies to dive-site records. This feature feels realistic for me because it can be built as a form-based system. A user could select a dive site and submit information such as current strength, visibility, marine life notes, personal reflections, and maybe an image path or upload reference. From my point of view, this is manageable because it follows a structure I can already imagine: HTML form input, Java backend processing, and then displaying the saved record on a results or detail page. What feels much less realistic is any feature that tries to automatically analyse these records and generate smart suggestions. That kind of system sounds interesting, but it goes beyond what I can confidently implement.
I had the same reaction when thinking about buddy matching. A true matching feature sounds useful, but if I think about building it with my current skill set, it quickly becomes too complicated. It would require more complex logic, probably more user conditions, and maybe even live communication. Right now, I think a more realistic replacement is much simpler: users can browse profile information, view someone’s tags or experience level, and use a provided contact method to reach out themselves. That still supports connection, but it matches my actual technical level much better.
For local dive shop information, the structure feels much easier to control. I can imagine a region selection page, followed by dive shop cards with a shop name, image, address, contact details, and short description. This feels technically achievable because the information is fixed and the page logic is relatively simple.
My main takeaway this week is that technical feasibility becomes much clearer when I stop describing features in general terms and start asking what I could realistically build with HTML and Java. Right now, I think I can support structured forms, profile display, basic searching or browsing, and simple information pages. What I cannot realistically build well are complex automated systems. Because of that, I think the project will be stronger if it focuses on features that are clear, structured, and manageable within my actual skill level.
